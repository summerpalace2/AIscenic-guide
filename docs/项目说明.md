# 项目说明：AI 数字人智能景区导览系统 —— 从零到部署的全栈 AI 项目实战

> **项目架构**：AI 端（Java Spring Boot, port 8081）+ 后端 Python（FastAPI, port 8000）+ 后端 Java（Spring Boot, port 8080）
> **AI 能力**：RAG 检索增强生成（DeepSeek + Milvus 向量库）、意图提取、对话记忆
> **数据库**：Python 端 MySQL（开发 SQLite）/ AI 端 Redis + Zilliz Cloud（Milvus）
> **认证机制**：JWT Bearer Token（access_token 1h + refresh_token 7d）
> **API 版本**：v1（前缀 /api/v1）
> **部署方式**：Docker 容器化 + docker-compose 多服务编排

## 第一篇：项目认知

### 第1章 项目总览

#### 1.1 赛题背景

国家大力推进文旅产业数字化转型，智慧景区建设已成为行业趋势。传统景区导览服务面临讲解资源不足、信息更新滞后、难以满足游客个性化与深度体验需求等痛点。本赛题旨在利用AI数字人技术，构建一个智能、互动、个性化的景区导览服务AI数字人，推动文旅服务的智能化升级。

#### 1.2 业务痛点

当前景区导览服务普遍存在以下痛点：

- **导游资源稀缺**：黄金周或旺季时，专业导游供不应求，游客体验大打折扣。
- **信息单向传递**：传统的录音导览设备内容固定，无法与游客互动，解答个性化问题。
- **缺乏情感连接**：冰冷的设备难以提供如真人导游般的亲切感和情感互动。
- **管理盲区**：景区管理者难以量化评估导游服务质量，无法精准获取游客真实反馈以优化服务。

#### 1.3 实现目标与实用价值

**实现目标**：开发一个具备多模态交互能力（语音、文本、表情）的AI数字人导游软件，能够为游客提供实时智能问答、个性化路线讲解、情感互动，并为景区管理方提供游客反馈分析。

**实用价值**：解决景区导游资源紧张问题，提升游客游览体验的沉浸感和满意度，帮助景区管理者数字化运营，降低人力成本，实现服务升级。

#### 1.4 技术栈全景

| 层面 | 技术选型 | 涉及课程领域 |
|------|---------|------------|
| AI 端 | Java 17 + Spring Boot 3 + Spring AI + Milvus + Redis | AI 工程、数据结构与算法 |

| 基础设施 | Docker + docker-compose + Git + Dev Containers | 操作系统、DevOps |

**涉及技术**：数字人建模与驱动、语音识别与合成、自然语言处理、大模型技术、多模态情感分析等。

#### 1.5 系统核心能力

从用户视角看，本系统主要解决三类问题：

- **游客如何获取信息**：通过文本/语音与 AI 数字人对话，获取景区知识、路线推荐、实时讲解。系统支持 SSE 流式响应和 WebSocket 全双工通信，实现毫秒级的实时交互。
- **管理员如何运营**：通过管理后台配置知识库（上传 Word/Excel/PDF 文档并自动同步到向量库）、设置数字人外观和音色、查看运营数据和分析报告。
- **系统如何保障体验**：RAG 检索增强生成确保回答准确率 >= 90%；两级排序（Milvus 向量检索 TOP30 + 重排序 TOP10）过滤噪声；WebSocket 流式推送保证感知延迟 < 5 秒。

#### 1.6 功能需求

**游客交互侧**：
1. 多模态交互：支持语音输入和文本输入，数字人能以语音、表情和口型同步的方式进行回答。
2. 智能问答与讲解：能够准确回答关于景区历史、文化、景点特色等常见问题。
3. 个性化推荐：可根据游客兴趣推荐不同的游览路线和讲解重点。

**管理后台侧**：
1. 知识库管理：管理员可上传、更新和维护景区的讲解词、文史资料、常见问题及答案等知识文档。
2. 数字人形象管理：可配置数字人的外观、服装、声音等，使其更贴合景区文化特色。
3. 游客感受度报告：系统能分析交互记录，生成游客关注点分析、情感趋势报告及服务建议。
4. 数据大屏概览：展示当日/本周服务人次、热门问答、游客满意度趋势等核心运营数据。

#### 1.7 非功能性需求

- **AI 模型**：需明确使用至少1个多模态大模型作为核心AI能力支撑
- **知识库**：需构建本地景区知识库，确保回答准确性和自然度
- **准确度**：景区事实性问答准确率不低于90%
- **自然度**：口型同步、语音合成、表情配合自然
- **响应延迟**：语音问答延迟 < 5秒
- **稳定性**：系统不应出现崩溃或长时间无响应

### AI 端应用启动流程

1. 独立微服务：AiApplication 是独立的 Spring Boot 应用，绑定端口 8081。
2. 加载环境变量：loadDotEnv() 在 Spring 启动前读取 .env 文件，注入 API 密钥。
3. DashScope 向量化服务初始化：AlibabaEmbeddingConfig 连接阿里云 DashScope text-embedding-v2。
4. Milvus 向量库连接：MilvusConfig 通过 gRPC 连接 Zilliz Cloud，启动时打印集合列表。
5. Tomcat 启动：监听 8081 端口，9 个 REST 端点就绪。

涉及知识范畴：操作系统、计算机网络、AI 工程

操作系统：JVM 进程独立运行（port 8081）、Spring 上下文生命周期
计算机网络：REST API（HTTP 远程调用）、gRPC（Milvus 高性能通信）、SSE（服务器推送）
AI 工程：RAG 流水线、向量检索、重排序算法、大模型 API 集成

#### 8.1 Engine / Session

```python
engine = create_engine(settings.DATABASE_URL, pool_pre_ping=True)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

def get_db():
    db = SessionLocal()
    try: yield db
    finally: db.close()
```

DATABASE_URL 开发环境为 sqlite:///./scenic_guide.db，生产环境切换 MySQL。pool_pre_ping=True 避免连接池断连。

#### 12.2 应用启动流程

Spring Boot 应用的启动顺序：

1. .env 加载 — ScenicGuideApplication.loadDotEnv() 在 SpringApplication.run() 之前执行，逐行读取 .env 文件，调用 System.setProperty(key, value)，确保 @Value 注解在依赖注入前拿到值。
2. Spring 上下文初始化 — @SpringBootApplication 自动扫描 com.ai.guide 包，注册所有 Bean。
3. Redis 健康检查 — RedisHealthCheck 的 @PostConstruct 方法 PING Redis，连接失败仅打印警告，不阻断启动。
4. Jackson 配置 — JacksonConfig 注册 JavaTimeModule，确保 LocalDateTime 序列化格式正确。
5. Tomcat 启动 — 监听端口 8080（可通过 server.port 配置）。
6. 控制器就绪 — ChatController、HistoryController、ImportController、PreferencesController 所有端点可访问。

#### 12.9 内置配置要点

1. 自定义 .env 加载：ScenicGuideApplication.loadDotEnv() 在 Spring 启动前读取 ai/.env（或根 .env），逐行 System.setProperty(key, value)。@Value("${...}") 注解通过系统属性获取值。

2. DeepSeek 伪装成 OpenAI：
```yaml
spring.ai.openai.base-url: https://api.deepseek.com
spring.ai.openai.api-key: ${DEEPSEEK_API_KEY}
```
使用 OpenAI SDK 调用 DeepSeek 模型，无需额外 SDK。

3. 向量化 API 兼容：DashScope 使用 OpenAI 兼容模式：
```yaml
dashscope:
  api-key: ${DASHSCOPE_API_KEY}
  base-url: https://dashscope.aliyuncs.com/compatible-mode/v1
```

4. Tika 版本固定 2.9.2：Spring AI 默认携带的 Tika 版本为 beta 版，手动排除并引入稳定版：
```xml
<exclusion><groupId>org.apache.tika</groupId><artifactId>tika-core</artifactId></exclusion>
```

5. Bean 覆盖允许：避免 OpenAI 自动配置与手动注册的 AlibabaEmbeddingConfig 冲突。
```yaml
spring.main.allow-bean-definition-overriding: true
```

6. Jackson 时间格式：
```java
@Bean
public ObjectMapper objectMapper() {
    return JsonMapper.builder()
        .addModule(new JavaTimeModule())
        .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS)
        .build();
}
```

#### 13.11 关于 AI Agent 辅助开发的说明

本项目使用 Prometheus（规划 Agent）+ Sisyphus（任务执行 Agent）等 AI Agent 进行辅助开发。这是目前业界前沿的开发方式，也是简历上的一个重要差异化亮点。

**什么是 AI Agent 辅助开发**

传统开发模式：需求分析 → 手写代码 → 手动测试 → 手动修复 BUG
本项目开发模式：需求描述 → AI Agent 生成方案 → AI Agent 编写代码 → AI Agent 执行测试 → 人工审核与修正

不再是你一行一行写代码，而是你作为"架构师 + 审核者"，AI Agent 作为"执行者"。你需要告诉 AI 做什么（需求）和怎么做是对的（验收标准），AI 负责具体的编码和测试执行。

**本项目中 AI Agent 参与的工作**

| 开发环节 | AI Agent 参与程度 | 具体工作 |
|----------|-------------------|----------|
| 需求分析与规划 | ★★★★★ | Prometheus 自动生成项目规划、模块划分、任务拆解 |
| 代码编写 | ★★★★☆ | 后端全部代码由 AI Agent 编写（100+ 文件，5000+ 行代码） |
| 测试编写 | ★★★★★ | pytest 测试用例由 AI Agent 自动生成并运行验证 |
| BUG 修复 | ★★★★☆ | AI Agent 分析错误日志 → 定位问题 → 修改代码 → 重新验证 |
| 数据库设计 | ★★★★☆ | ORM 模型定义、字段类型选择、迁移脚本编写 |
| 架构决策 | ★★★☆☆ | AI 提供方案对比和选型建议，人工最终决策 |
| 技术文档 | ★★★★★ | 本文档全部由 AI Agent 根据代码结构自动生成 |
