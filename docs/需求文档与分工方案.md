# AI数字人智能景区导览系统 —— 需求文档与分工方案

---

## 一、项目概述

### 1.1 项目背景
国家大力推进文旅产业数字化转型，传统景区导览面临导游资源稀缺、信息单向传递、缺乏情感连接、管理盲区等痛点。本项目利用AI数字人技术，构建一个7×24小时在线的智能导览系统。

### 1.2 项目目标
开发一个具备多模态交互能力（语音、文本、表情）的AI数字人导游系统，包含**游客交互端**和**管理后台**两大模块，实现逼真数字人交互、精准景区知识问答和深度游客洞察。

### 1.3 团队角色
| 角色 | 职责范围 |
|------|----------|
| **AI端** | 大模型集成、语音/视觉处理、数字人引擎、知识库RAG、情感分析 |
| **后端** | API服务、知识库管理、数据存储、用户系统、数据聚合分析 |
| **前端** | 游客交互界面、管理后台界面、数字人展示、数据可视化 |

## 三、非功能性需求

| 编号 | 类型 | 要求 |
|------|------|------|
| NF1 | AI能力 | 至少使用**1个多模态大模型**作为核心AI能力支撑 |
| NF2 | 知识库准确率 | 事实性问答准确率 **≥90%**（专家基于标准测试集评测） |
| NF3 | 自然度 | 口型同步、语音合成、表情配合自然流畅（专家评估演示视频） |
| NF4 | 响应延迟 | 语音问答响应延迟 **<5秒** |
| NF5 | 稳定性 | 系统不出现崩溃或长时间无响应 |
| NF6 | GPS弱信号 | （可选）考虑GPS信号弱或难以定位场景的降级方案 |
| NF7 | 安全性 | 接口鉴权、数据传输加密、防注入攻击 |
| NF8 | 可扩展性 | 前后端分离架构，AI模块可独立升级替换 |

## 六、模块依赖关系

```
                    ┌──────────────┐
                    │   AI端核心    │
                    │ (大模型+RAG)  │
                    └──────┬───────┘
                           │ 提供AI能力
                    ┌──────▼───────┐
                    │   后端服务    │
                    │(API+数据+存储)│
                    └──────┬───────┘
                           │ 提供接口
              ┌────────────┼────────────┐
              │            │            │
     ┌────────▼───┐  ┌─────▼──────┐  ┌─▼──────────┐
     │  游客前端   │  │  管理前端   │  │  数字人渲染  │
     │ (移动端H5)  │  │ (Web后台)  │  │ (前端驱动)   │
     └────────────┘  └────────────┘  └─────────────┘
```

**依赖说明：**
1. **AI端** 是最底层能力提供方，后端调用AI端服务接口
2. **后端** 整合AI能力 + 业务逻辑，对前端暴露统一RESTful API + WebSocket
3. **前端** 游客端和管理端共用同一后端API，但为不同入口

## 七、项目文件结构

根据团队分工与模块依赖关系，项目采用 monorepo 风格管理，按技术栈划分为 `ai/`（Spring AI + Java）、`backend/`（FastAPI + Python）、`frontend/`（Vue3 + TypeScript）三大独立子目录，各端在各自目录下并行开发。

### 7.1 概览（摘要）

```
ai-guide-system/
│
├── ai/                       # AI端 — Spring AI (Java + Maven)
│   ├── src/                  #   源码 (main/java + test/java)
│   ├── pom.xml               #   Maven依赖管理
│   └── Dockerfile
│
├── backend/                  # 后端 — FastAPI (Python)
│   ├── app/                  #   应用主目录（API / 模型 / 服务）
│   ├── tests/                #   测试
│   ├── requirements.txt      #   Python依赖
│   ├── Dockerfile
│   └── docker-compose.yml    #   服务编排
│
├── frontend/                 # 前端 — Vue3 + TypeScript
│   ├── tourist/              #   游客交互端（移动端H5）

│   └── shared/               #   跨端共享类型与工具
│
├── docs/                     # 项目文档（API / 架构 / 部署）
```

**目录设计原则：**
1. **三端隔离**：`ai/`、`backend/`、`frontend/` 三个顶级目录对应三个团队，互不干扰，可独立构建部署
2. **子模块对齐**：子目录结构与分工方案（第五章）一一对应，每个团队按自己负责的模块组织代码
3. **API层独立**：`backend/app/api/v1/` 每个资源一个文件，符合RESTful最佳实践，便于接口文档生成
4. **前端双入口**：`frontend/tourist/`（游客H5）和 `frontend/admin/`（管理后台SPA）独立构建，共享 `shared/` 类型定义
5. **测试就近**：各端 `tests/` 目录紧贴源码，不另建独立仓库
6. **文档集中**：统一 `docs/` 目录存放架构、API、部署文档

## 九、协作规范

1. **接口先行**：后端先出接口文档，前后端按文档并行开发

5. **测试驱动**：AI端写评测脚本，后端写单元/集成测试，前端写组件测试
